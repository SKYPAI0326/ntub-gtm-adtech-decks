Step 02 · 內容類型與 KPI 軟硬度

內容類型左欄，KPI 軟硬右欄

Content · 內容與來源
三層該放什麼
TOFU：SEO 長尾、YouTube Shorts、IG Reels、Meta 品牌廣告。
MOFU：比較文、開箱、EDM、再行銷廣告。
BOFU：試用、限時折扣、品牌字、購物車挽回信。
常見坑：把 TOFU 流量直接導到 BOFU 頁面，轉換率必慘。中間要有 MOFU 加熱。
KPI · 軟到硬
指標越往下越硬
TOFU：impressions、新訪客數、平均停留。
MOFU：訂閱數、下載數、產品頁停留。
BOFU：加入購物車、結帳開始、實際下單、LTV。
釘 KPI：每層都釘指標才看得出哪一段在漏水。只看最終轉換率找不到瓶頸。

Real Case · 漏斗找出結帳卡關

從月營收 300 萬到420 萬

美妝品牌原本只追蹤 purchase，不知道哪段在漏水。
埋完整漏斗兩週後找到瓶頸：結帳頁。

關鍵發現：begin_checkout → add_payment_info 轉換率只有 38%（產業均值 60%）。
追查後是結帳頁必填欄位太多 + 手機鍵盤遮提交鈕。改版後該段拉到 62%。
問題本質
不是漏斗有問題，是沒漏斗所以看不到問題。
visibility
給講師的話
漏斗定義要跟業務對齊——B2C / 訂閱 / SaaS 的 begin_checkout 時機都不同。

Step 02 · 命名前先查 GA4 推薦清單

有推薦事件就用推薦

Recommended · 推薦事件
先查官方清單
GA4 推薦事件：purchase、add_to_cart、generate_lead、begin_checkout、view_item...
用推薦的事件，GA4 後台會自動跑出對應報表，省下大量設定。
查 developers.google.com/analytics/devguides/collection/ga4/reference/events
Custom · 自訂事件
沒推薦再自訂
沒推薦的才自訂。一律 snake_case、動詞_名詞、英文、不超過 40 字元。
Meta / Ads / TikTok 標準事件可能用 PascalCase——GTM Tag 內轉換即可，不要讓 dataLayer 跟著亂。
NG：BtnClick · 購買成功 · click-button
OK：submit_contact_form · download_brochure

Step 03 · value 與 currency 最常忘

沒填 value，ROAS 變零

硬規則：所有跟金錢相關的事件（purchase、add_to_cart、begin_checkout）必填 value 與 currency。沒填 value 的 purchase 在廣告平台歸因會被當作 $0 轉換。currency 必須用 ISO 4217（TWD/USD/JPY）。
B2B SaaS 案例
某 HRTech 不能照搬電商：demo_requested → demo_scheduled → demo_attended → proposal_sent → contract_signed，每事件帶 deal_size + industry + company_size。
5 events
解鎖的問題
行銷終於能回答：「哪個產業的 demo 轉簽約率最高？」答案：金融業 31%，高於製造業 12%。
31% vs 12%
B2B 坑
事件命名要跟 Sales 的 CRM stage 對齊（如 HubSpot Deal Stage），否則 GA4 漏斗跟 CRM 漏斗會說兩套故事。
align-CRM

Real Case · 三家代理商亂拼 UTM

規範前 vs 規範後，同一份報表

Before · 沒規範
三家同投 FB
A 用 utm_source=facebook
B 用 utm_source=FB
C 用 utm_source=Facebook
—— GA4 變成三個 source，歸因報表一片混亂。
月會無法回答：「FB 整體 ROAS 多少？」
After · 規範 + 工具
UTM 產生器強制
統一規範（小寫、底線、無空格），重發給代理商。Looker Studio 用 REGEX + LOWER 把舊資料統整。
產生器做成下拉選單——人手打不進去就不會出錯。
規範 = 規則 + 工具 + 流程，三者缺一不可。寫 PDF 沒人看。
